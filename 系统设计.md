# Web3Proof V1.0 系统设计文档

> 本文档在《部署方案.md》基础上，补充数据库、API、智能合约、前端、部署、安全与验收等可落地设计细节。V1.0 已调整为“基于区块链的学生成果存证与核验平台”。

---

# 一、设计原则

| 原则 | 说明 |
|------|------|
| 链上存证、链下索引 | 关键摘要与哈希上链；详情存 IPFS；数据库做查询索引与业务状态 |
| 用户自签交易 | 学生通过钱包签名完成 DID 登记与成果存证，后端不托管用户私钥 |
| 外部语义改造、内部兼容 | 前端与文档展示学生成果业务；API、数据库表、合约名暂不重命名 |
| 最小可用 | V1.0 聚焦学生、管理员、核验方三类角色 |
| 前后端分离 | 前端 React，后端 Go REST API，合约 Hardhat 独立工程 |

# 五、智能合约设计

合约名称保持不变，避免重新部署和迁移。

## 5.1 DIDRegistry.sol

负责学生链上身份登记。

```solidity
registerDID(string did, string github, string metadataCID)
updateProfile(string github, string metadataCID)
getProfile(address owner)
```

## 5.2 ProjectRegistry.sol

负责成果材料哈希存证。内部结构仍叫 `Project`，展示语义为“成果”。

```solidity
addProject(string name, string ipfsCID, bytes32 contentHash, string githubUrl, address contractAddr)
getProject(uint256 projectId)
getProjectsByOwner(address owner)
verifyProject(uint256 projectId, bytes32 expectedHash)
```

## 5.3 SkillSBT.sol

负责发放不可转让成果认证凭证。内部变量 `skillType` 对应数据库 `skill_categories.id`，展示为成果类别。

```solidity
mint(address to, uint256 skillType, string metadataCID)
burn(uint256 tokenId)
locked(uint256 tokenId)
verify(address owner, uint256 skillType)
```

## 5.4 Reputation.sol

负责可信评分上链同步。

```solidity
updateScore(address user, uint256 projectScore, uint256 certScore, uint256 activityScore)
getScore(address user)
```

# 六、评分算法

| 维度 | 满分 | 规则 |
|------|------|------|
| 成果存证 | 500 | 每个已确认链上成果 +100，上限 5 个 |
| 认证凭证 | 300 | 每个有效 SBT +100，上限 3 个 |
| 资料完整度 | 200 | 登录活跃、近期成果、近期申请、资料完整、DID 已注册 |

综合评级：

| 等级 | 分数区间 |
|------|------|
| A | 800 - 1000 |
| B | 600 - 799 |
| C | 400 - 599 |
| D | 0 - 399 |

# 七、核心业务流程

## 7.1 学生成果存证流程

```text
1. 学生填写成果名称、说明、成果链接
2. POST /projects，后端上传元数据到 IPFS
3. POST /projects/:id/prepare-chain，返回合约调用参数
4. 前端调用 ProjectRegistry.addProject()
5. POST /projects/:id/confirm-chain，后端校验交易并更新为 confirmed
6. 重新计算可信评分
```

## 7.2 成果认证与 SBT 发放流程

```text
1. 学生选择成果类别并提交材料说明
2. 管理员审核认证申请
3. 审核通过后准备 mint 参数
4. 管理员钱包调用 SkillSBT.mint()
5. 后端确认交易并写入 sbt_records
6. 重新计算可信评分
```

## 7.3 核验方核验流程

```text
1. 核验方钱包登录并注册单位/机构资料
2. 输入学生钱包地址
3. GET /verify/:address 聚合数据库与链上校验结果
4. 展示 DID、成果数量、认证凭证、可信评分与链上记录
```

# 十、安全设计

| 风险 | 措施 |
|------|------|
| 伪造登录 | nonce 一次性 + 钱包签名校验 |
| 材料篡改 | IPFS CID + 内容哈希上链 |
| 凭证冒用 | SBT 不可转让，与学生钱包绑定 |
| 越权访问 | JWT + role 中间件 + 资源 owner 校验 |
| 私钥泄露 | 用户私钥不进后端；管理员/后端钱包仅保留测试资产 |
| 重复发证 | 数据库唯一约束 + 合约 `hasSkillType` 限制 |

# 十一、测试与验收

## 11.1 测试清单

| 类型 | 内容 |
|------|------|
| 合约测试 | DID 注册、成果存证、SBT 不可转让、可信评分更新 |
| 后端测试 | 钱包验签、IPFS mock、成果认证、核验报告 |
| 前端测试 | 钱包连接、成果创建、链上交易、审核发证、核验报告 |
| E2E | 学生完整流程 + 管理员审核 + 核验方查询 |

## 11.2 V1.0 验收标准

- [ ] MetaMask 钱包可登录并获取 JWT
- [ ] 学生可注册 DID 并在链上可查
- [ ] 学生可完成至少 1 个成果材料 IPFS + 链上存证
- [ ] 管理员可审核并发放 SBT 认证凭证
- [ ] 可信成果档案正确展示 DID、成果、SBT、可信评分
- [ ] 核验方可查看核验报告并显示评级
- [ ] 新路由 `/achievements`、`/certificates`、`/portfolio/:address` 可访问
- [ ] 旧路由 `/projects`、`/certifications`、`/resume/:address` 仍兼容可访问
