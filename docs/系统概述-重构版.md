# Web3Proof 系统概述重构版

## 一、项目定位

Web3Proof 是一个基于区块链的数字作品与学生成果存证、认证、核验平台。系统面向学生、个人创作者、教师或管理员、第三方核验机构等多类用户，提供作品上传、哈希存证、链上交易记录、认证审核、凭证生成、公开核验和可信档案展示等能力。

本项目不定位为虚拟币交易平台，也不提供代币发行、充值、提现、NFT 买卖或二级市场功能。区块链在系统中的作用是保存不可篡改的存证摘要、凭证记录和核验依据，业务数据、文件资料、审核记录和用户信息仍保存在链下服务器与数据库中。

推荐项目题目：

> 基于区块链的数字作品存证与可信核验平台的设计与实现

也可以保留学校场景，题目写作：

> 基于区块链的学生成果存证与第三方核验平台的设计与实现

## 二、建设背景

在课程项目、竞赛证书、实习证明、科研成果、设计作品、论文材料、摄影图片、代码项目等数字资料的流转过程中，常见问题包括：

- 文件容易被复制、篡改或冒用；
- 作品创建时间和归属关系难以自证；
- 证书、截图、项目经历依赖人工核验；
- 第三方单位难以快速判断材料真实性；
- 传统数据库记录存在被后台修改的可能；
- 用户需要一个可公开展示、可验证的可信成果档案。

区块链具有公开可查、交易可追溯、数据不可随意篡改等特点，适合用于保存文件哈希、作者地址、存证时间、认证凭证等关键摘要信息。通过“链上存摘要、链下存文件和业务数据”的方式，可以兼顾可信性、性能、成本和隐私安全。

## 三、总体目标

系统目标是构建一个可部署、可演示、可扩展的区块链存证 DApp 平台，实现以下能力：

- 用户可以使用账号密码或钱包登录系统；
- 用户可以在不同身份角色之间切换工作台；
- 用户上传作品或成果文件后，系统生成文件哈希；
- 系统将哈希、作者地址、存证编号和时间写入智能合约；
- 用户可以下载电子存证证书；
- 管理员可以审核成果认证申请并发放 SBT 认证凭证；
- 第三方核验方可以通过二维码、证书编号、钱包地址或文件哈希验证材料真实性；
- 系统支持 Sepolia 测试网演示，也预留私有链、联盟链或自建节点切换能力；
- 系统可以部署到 Linux 服务器，通过 Nginx、Docker、MySQL 对外提供服务。

## 四、核心用户角色

### 4.1 普通用户 / 学生 / 创作者

普通用户是平台的主要使用者，可以上传自己的数字作品、课程成果、项目材料、证书证明等内容。

主要功能：

- 注册、登录、绑定钱包；
- 维护个人资料；
- 上传作品或成果文件；
- 计算文件哈希并提交链上存证；
- 查看个人存证记录；
- 申请官方认证；
- 下载电子存证证书；
- 生成个人可信档案主页；
- 分享公开核验链接或二维码。

### 4.2 认证管理员

认证管理员负责审核用户提交的认证申请，确认材料是否完整、真实、合规。

主要功能：

- 查看待审核认证申请；
- 查看用户提交的作品、附件和链上记录；
- 通过或驳回认证申请；
- 填写审核意见；
- 发放不可转让的 SBT 认证凭证；
- 管理作品分类、成果分类和系统公告；
- 查看平台统计数据。

### 4.3 第三方核验方

第三方核验方可以是企业 HR、教师、竞赛组织方、学校部门或外部审核机构。该角色主要用于查询和验证材料。

主要功能：

- 注册核验方身份；
- 切换到核验工作台；
- 通过证书编号、钱包地址、交易哈希、文件哈希查询存证；
- 上传待验证文件并重新计算哈希；
- 查看链上交易、区块高度、合约地址；
- 生成核验报告；
- 保存历史核验记录。

### 4.4 超级管理员

超级管理员负责平台配置、权限分配、数据监控和系统维护。

主要功能：

- 管理用户账号；
- 分配用户角色；
- 审核核验机构资质；
- 配置链网络参数；
- 配置合约地址；
- 查看系统运行日志；
- 查看链上事件同步状态；
- 管理敏感内容、举报与申诉。

## 五、用户角色切换设计

系统支持一个用户拥有多个角色。用户登录后，可以在页面顶部或侧边栏切换当前工作身份。

示例：

```text
当前账号：0xA12...89F
可用身份：创作者 / 核验方 / 管理员
当前工作台：创作者工作台
```

角色切换规则：

- 用户默认拥有普通用户角色；
- 用户可以申请成为核验方；
- 管理员角色由超级管理员分配；
- 切换角色只改变当前可访问的页面、菜单和操作权限；
- 后端每个接口仍需要根据 JWT 和角色进行权限校验；
- 数据库保存用户拥有的角色列表和当前默认角色。

建议角色字段设计：

```text
users
- id
- username
- wallet_address
- default_role
- status
- created_at

user_roles
- id
- user_id
- role_code
- enabled
- created_at
```

角色编码建议：

| 角色名称 | 角色编码 | 说明 |
|---|---|---|
| 普通用户 / 创作者 | `creator` | 上传作品、发起存证、申请认证 |
| 核验方 | `verifier` | 查询档案、验证文件、生成报告 |
| 认证管理员 | `auditor` | 审核认证申请、发放 SBT |
| 超级管理员 | `admin` | 系统配置、用户管理、权限管理 |

## 六、核心业务流程

### 6.1 作品存证流程

```text
用户登录
  -> 上传作品文件
  -> 系统计算 SHA-256 哈希
  -> 文件保存到本地服务器、MinIO 或 IPFS
  -> 后端生成存证编号
  -> 调用智能合约写入作品哈希和作者地址
  -> 后端记录交易哈希、区块号、合约地址
  -> 生成电子存证证书
  -> 用户下载证书或分享核验链接
```

链上保存：

- 存证编号；
- 文件哈希；
- 作者钱包地址；
- 元数据 CID 或摘要；
- 存证时间；
- 存证状态。

链下保存：

- 原始文件或文件访问地址；
- 作品标题；
- 作品描述；
- 分类标签；
- 用户资料；
- 审核记录；
- 证书 PDF；
- 操作日志。

### 6.2 认证审核流程

```text
用户选择已存证作品
  -> 提交认证申请
  -> 上传补充证明材料
  -> 管理员审核材料
  -> 审核通过
  -> 管理员调用 SBT 合约发放认证凭证
  -> 系统同步链上事件
  -> 用户档案展示认证状态
```

认证状态建议：

| 状态 | 说明 |
|---|---|
| `draft` | 草稿 |
| `pending` | 待审核 |
| `approved` | 审核通过 |
| `rejected` | 审核驳回 |
| `minted` | 已发放链上凭证 |
| `revoked` | 已撤销 |

### 6.3 文件核验流程

```text
核验方上传待验证文件
  -> 系统计算文件哈希
  -> 查询数据库是否存在该哈希
  -> 查询智能合约确认链上记录
  -> 对比作者地址、存证时间、交易哈希
  -> 输出核验结果
  -> 生成核验报告
```

核验结果示例：

```text
核验结论：通过
文件哈希：0x...
存证编号：CERT-2026-000001
作者地址：0x...
交易哈希：0x...
区块高度：...
存证时间：2026-06-26 10:30:00
说明：该文件哈希与链上存证记录一致，文件内容未发现篡改。
```

### 6.4 公开档案展示流程

```text
用户开启公开档案
  -> 系统生成公开链接
  -> 访问者查看用户公开作品
  -> 查看认证凭证和可信评分
  -> 点击单个作品查看链上存证详情
```

公开档案不展示隐私文件原文，只展示用户主动公开的信息、作品摘要、认证状态和核验入口。

## 十、智能合约设计

### 10.1 EvidenceRegistry 存证合约

负责记录作品或成果的链上存证信息。

核心字段：

```solidity
struct Evidence {
    uint256 id;
    bytes32 evidenceNoHash;
    bytes32 fileHash;
    address owner;
    string metadataURI;
    uint256 createdAt;
    bool revoked;
}
```

核心方法：

```solidity
createEvidence(bytes32 evidenceNoHash, bytes32 fileHash, string metadataURI)
getEvidence(uint256 id)
getEvidenceByHash(bytes32 fileHash)
verifyEvidence(bytes32 fileHash)
revokeEvidence(uint256 id)
```

### 10.2 CredentialSBT 认证凭证合约

负责发放不可转让的认证凭证。

核心方法：

```solidity
mintCredential(address to, uint256 evidenceId, string tokenURI)
revokeCredential(uint256 tokenId)
credentialOf(uint256 tokenId)
hasCredential(address owner, uint256 evidenceId)
```

设计要点：

- SBT 不允许转让；
- 只有管理员地址可以发放；
- 凭证与存证记录绑定；
- 凭证可以被管理员撤销；
- tokenURI 指向认证元数据。

### 10.3 DIDProfile 身份合约

用于绑定用户链上身份摘要。

核心方法：

```solidity
registerProfile(string did, string metadataURI)
updateProfile(string metadataURI)
getProfile(address owner)
```

## 十三、部署与链网络说明

### 13.1 开发和答辩环境

开发阶段推荐使用 Sepolia 测试网。

说明口径：

> 系统在开发和测试阶段使用 Ethereum Sepolia 测试网，用于验证智能合约部署、链上存证、交易查询和钱包交互流程。Sepolia 不承载真实资产交易，仅用于测试和演示。

### 13.2 国内服务器部署

国内服务器部署建议：

```text
Linux 服务器
  |
  |-- Nginx：反向代理前端和后端接口
  |-- Frontend：React 静态文件
  |-- Backend：Go API 服务
  |-- MySQL：业务数据库
  |-- Redis：缓存与任务状态
  |-- MinIO：文件对象存储
  |-- Chain Adapter：连接 Sepolia RPC 或私有链节点
```

注意事项：

- Sepolia 可以作为测试链使用，但国内服务器访问海外 RPC 可能不稳定；
- 生产部署建议保留私有链、联盟链或自建节点适配能力；
- 不要将用户隐私、手机号、身份证号、原始文件内容直接写入链上；
- 不提供虚拟币交易、NFT 买卖或代币融资功能；
- 对外运营时需要关注区块链信息服务备案、内容审核和用户协议。

## 十四、系统亮点

- 将文件哈希写入区块链，降低作品被篡改后无法自证的问题；
- 支持账号登录和钱包登录，兼顾普通用户体验与 DApp 特性；
- 支持用户角色切换，一个账号可同时是创作者、核验方或管理员；
- 采用 SBT 表达不可转让的认证凭证，比普通 NFT 更适合证书场景；
- 提供电子证书和二维码核验，适合真实上线演示；
- 链上只保存摘要，链下保存业务数据，兼顾隐私和性能；
- 支持 Sepolia 测试网，也预留生产环境私有链或联盟链切换；
- 功能覆盖前端、后端、数据库、智能合约、Linux 部署和软件测试，适合作为毕业设计或课程综合项目。

## 十七、项目边界说明

本系统提供的是数字作品和成果材料的技术存证、认证辅助和核验辅助能力，不直接等同于司法鉴定结论或版权登记证书。系统可以证明某个文件哈希在某个时间点已经被某个钱包地址提交到链上，但作品权属仍可能需要结合原始创作过程、平台审核、第三方证明和法律程序进一步确认。

因此，系统更准确的定位是：

> 为数字作品和成果材料提供低成本、可追溯、可公开核验的可信存证基础设施。
